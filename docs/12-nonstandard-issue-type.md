# Тип заявки «Нестандарт» (вместо «Другое» в списке)

Вариант **без** программируемых сценариев (тариф ниже Expert 50+).

## Зачем

Сейчас «Другое» в `typical` — самый короткий путь. Жёстко «чекбокс → обязательное поле» Okdesk без автоправил не умеет.  
Отдельный **тип заявки** позволяет сделать уточнение обязательным **только** для нестандарта (обязательность атрибута привязана к типам / к статусу «Решена» для этого типа).

## Схема

| Обычная заявка | Нестандарт |
|----------------|------------|
| Тип: как сейчас (incident, service, …) | Тип: **Нестандарт** (новый) |
| `typical` — только нормальные пункты каталога (**без** «Другое») | Поле **«Суть нестандарта»** обязательно при «Решена» |
| Баллы, способ решения — как обычно | **`solution_method` — всё равно из каталога способов** (не «неизвестно что сделали») |

«Нестандарт» = нестандартная **проблема**, не отсутствие способа. Целевой список способов как раз закрывает бывшее «Другое» в solution.


## Настройка в Okdesk (сами)

1. **Типы заявок** → создать тип, код например `nonstandard`, название «Нестандарт».
2. Убрать значение **«Другое»** из списка `typical` (или оставить только для старых заявок / не показывать на новых типах — по возможностям привязки атрибута к `issue_type_codes`).
3. Новый атрибут, код например `nonstandard_detail`, название «Суть нестандарта» (строка или текст).
4. Привязать атрибут **только к типу** `nonstandard` (если в настройках атрибута есть список типов — у вас так уже сделано для других полей).
5. Сделать атрибут **обязательным** при входе в статус «Решена» (и при необходимости «Закрыта») — для этого типа он будет единственным «лишним» обязательным полем.
6. Для типа «Нестандарт» решить по остальным обязательным полям:
   - `typical` — снять обязательность **или** дать короткий список (например только «Нестандарт»);
   - `solution_method` — оставить обязательным, но без единственного «Другое» как отговорки (по возможности убрать «Другое» и оттуда для этого типа, либо смириться и смотреть текст уточнения).

## Правила для команды

- Нестандарт = **реально нет** подходящего пункта в каталоге (или смесь, которую нельзя отнести).
- В «Суть нестандарта» — одна фраза: что случилось / что сделали (не «-»).
- Раз в месяц: отчёт по типу `nonstandard` — доля от нативного контура (без MFC), топ формулировок → кандидаты в каталог.
- MFC-перенос **не** гонять через «Нестандарт» (оставить их процесс / тип как сейчас).

## Связь с баллами

- Баллы (`ticket_weight`) на нестандарте — по-прежнему; позже: норма «базовый вес по тексту уточнения» или отдельная вилка + аудит.
- Высокая доля типа «Нестандарт» у сотрудника (без MFC) — тот же сигнал, что раньше давало «Другое».

## Плюсы / минусы

| Плюсы | Минусы |
|-------|--------|
| Работает без Expert | Лишний шаг: сменить тип |
| Обязательное уточнение только когда нужно | Часть людей будет забывать сменить тип и ткнет «ближайший» пункт |
| Удобно фильтровать и анализировать | Нужно обучить команду 1 раз |
| Убирает «Другое» из основного списка | Старые заявки с «Другое» в истории останутся |

## Внедрение по шагам

1. Создать тип + поле, **пока** не убирать «Другое» из списка (параллельно 2 недели).
2. Объявить команде: новые нестандарты — только через тип.
3. Убрать «Другое» из `typical`.
4. Через месяц — разбор формулировок «Суть нестандарта» → донастройка каталога.

## Что проверить у себя в UI

В карточке доп. атрибута: есть ли список **типов заявок**, для которых поле показывается, и обязательность **при смене статуса**.  
Если обязательность только глобальная «для всех типов» — тогда либо тип «Нестандарт» + отдельный набор атрибутов, либо уточнить у Okdesk, как у вас на тарифе вешается required на статус.
