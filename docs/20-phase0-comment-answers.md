## Ответы по разделам

### §1 / approve.3 — базы T5/C15/S30 и «даже 60?»

**Да:** к каждой типовой проблеме в новом списке проставляем **одну базу**:

| Код | Баллы | Смысл |
|-----|------:|-------|
| T5 | 5 | типовое |
| C15 | 15 | среднее / консультация |
| S30 | 30 | сложная настройка |

**60 — не отдельная «база typical».** При чекбоксе выезда баллы заявки = **только 60**. Без выезда 60 может получиться как `база×N` или `S30+осложнение`.

**Следующий шаг:** таблица «typical → T5/C15/S30» — готовый лист: [22-typical-base-weights.md](22-typical-base-weights.md).

### §2 — что такое Expert?

**Expert 50+** — тариф Okdesk с автоправилами вида «если поле A = X, то поле B обязательно».

У вас тарифа нет → нельзя автоматически требовать «если раскатка — то N обязательно». Поэтому: **регламент + аудит + отчёты**, не магия в Okdesk.

### §3 — два поля L1 и L2?

**Нет.** В Okdesk заводите **два** дополнительных атрибута:

| # | Название | Тип атрибута (как в Okdesk) | Значения |
|---|----------|-----------------------------|----------|
| 1 | Осложнение | **Значение из списка** | `+15`, `+30` |
| 2 | Описание осложнения | **Строка** | текст причины |

**Правило начисления:** надбавка только если выбрано значение из списка **и** описание не пустое.  
Если уровень есть, а описание пустое → **0 баллов** за осложнение + **красная** строка в отчёте премии/аудита.  
Без Expert нельзя заставить Okdesk требовать описание автоматически — ловим формулой и отчётом.

### §4 — поле N: список или ручной ввод? Риск 10→100

**Числовое поле** (ручной ввод) + **список объектов в описании** для сверки.

Защита от «100 вместо 10»:

1. В описании/таблице видно реальное число строк → при разборе сверяем с N.  
2. Аудит: N ≥ порога (например 20) или `база×N` ≥ 100 → в отчёт.  
3. Опционально позже: потолок N на типикал (если Okdesk умеет) или проверка в отчёте премии до выплаты.

Полностью исключить опечатку без Expert нельзя — ловим сверкой и аудитом.

### §6 / approve.10 — MFC: без компенсатора

**Обновление решения:** компенсатор «МФС. Заявки, звонки» **не планируем**.

Цель: убрать пустые техзаявки и компенсирующие; массовый перенос — **удобная форма с выбором типа (typical) и способа** на каждую заявку → обычная норма (база × N при пакете).

В документах формулировку «bulk=0 + компенсатор» меняем на этот курс.

### §7 — что конкретно делаем?

Три слоя (не путать):

| Где | Что |
|-----|-----|
| **Okdesk** | Списки typical/solution; поле-**список** осложнения + строка; N; «Нестандарт»; выезд = чекбокс → **только 60** в формуле |
| **Регламент** | Когда ставить +15/+30, как заполнять N, кто целевой по расследованиям, сегменты конвейера |
| **Формула премии** | С даты X: выезд → только 60; иначе `(база×N + осложнение) × дежурство`; до X — только тень |

§7 (конвейер / расследования / SLA) — в основном **регламент + маршруты/очереди**, не отдельная «магическая» формула в Okdesk. Рубли сегментами режем позже; сначала каталог и нормы.

### §8 — выезд = только 60

**Исправлено по вашему уточнению.** Выезд — не надбавка к typical и не «база + 60».

```
если чекбокс выезда:
  баллы = только 60  (×2 в дежурство → 120)
  typical, N, осложнение в сумму НЕ входят
иначе:
  баллы = (база × N + осложнение) × (1|2)
```

Раньше в ответе был неверный пример «S30 + 60 = 90». При выезде правильно: **только 60**.
