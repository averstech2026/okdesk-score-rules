# Ревизия фазы 0: что было → что станет

Документ для согласования перед внедрением полей в Okdesk (фаза 1).  
Статус: **черновик на утверждение** (правила осложнения и выезда обновлены).  
Рабочий чеклист: [18-phase0-decisions.md](18-phase0-decisions.md). План: [17-implementation-plan.md](17-implementation-plan.md).

## 1. Модель баллов и премии

| | Было | Станет |
|---|------|--------|
| Источник баллов | Инженер выбирает число в `ticket_weight` (select **5…1000**) | Баллы считает **правило**: база из typical + пакет × N + осложнение + выезд, затем ×2 дежурства |
| Ориентиры 5 / 15 / 30 | Неформальные | Коды **T5=5 / C15=15 / S30=30** — единственные базы |
| Тяжёлый кейс vs 6 простых | Часто «ставлю 60 сам» или недооценка | База + **осложнение +15 или +30** |
| Пакет (раскатка на 40 касс) | Часто одна большая цифра «на глаз» (300–1000) | **`база × N`**, N видно в заявке |
| Выезд | Зоны 30 / 60 / 120 или «база + зона» | Чекбокс → **только 60** (typical/осложнение не суммировать) |
| Курс | 1 балл = 15 ₽ (факт) | Без изменений |
| Статьи премии | Сумма баллов; дежурства отдельным циклом ~1,5 нед | **Две статьи:** обычные заявки / дежурство; дежурство сверять с **табелем** |
| Когда платим по-новому | — | Только с **даты X** (после тени 4–6 нед) |

### Формула «станет» (v1)

```
если выезд (departure):
  баллы = 60 × (2 если дежурство, иначе 1)
  // typical / N / осложнение НЕ суммируются

иначе:
  баллы = (база(T5|C15|S30) × N
           + осложнение только если уровень И непустое описание
          ) × (2 если дежурство, иначе 1)

премия_₽ = сумма_баллов × 15
```

## 2. Поля Okdesk

| Поле | Было | Станет |
|------|------|--------|
| `ticket_weight` | Свободный select 5…1000 — главный источник премии | До даты X — как сейчас; после X — **заморозка** / служебный расчёт по правилам |
| `typical` | Обязательный select; много «Другое» / «Доп. настройки» | Обязательный; **сжатый каталог** + явные раскатки/хвост; источник **базы** |
| `solution_method` | Multiselect; «Другое» доминирует | Сжатый список; вместе с typical; для аудита, **не** источник баллов v1 |
| Осложнение | **Нет** | **Два** атрибута: «Значение из списка» (+15/+30) + «Строка»; надбавка только если **оба** заполнены, иначе 0 + красный в отчёте |
| `departure` | Checkbox + зона 30/60/120 | Checkbox → баллы заявки = **только 60** |
| `watch` | Checkbox ×2 вручную | Остаётся; истина — **табель**; статья «дежурство» отдельно |
| N объектов | Нет явного поля; объём прячут в баллах | **Атрибут «Число объектов»** + список объектов в описании |
| Тип «Нестандарт» | Нет (дыра = «Другое») | Тип + поле **«Суть»**; не для MFC-bulk |
| Expert-автоправила | Нет (тариф) | По-прежнему нет → **регламент + аудит** |

## 3. Осложнение (новый слой)

| | Было | Станет |
|---|------|--------|
| Учёт тяжести сверх нормы | Свободная цифра в баллах | Поле-**список**: **+15** или **+30** + текст описания |
| Видно ли почему тяжело | Часто нет | Текст описания осложнения |
| Суммирование уровней | — | Нет: одно значение в списке |
| На выезде | — | Осложнение **не влияет** на баллы (выезд = только 60) |

**Не путать:** осложнение ≠ объём пакета (N) ≠ выезд ≠ дежурство.

## 4. Типовые проблемы (`typical`)

### 4.1 Слияния дублей

| Было | Станет |
|------|--------|
| Оплата ДУЭТ + отдельно банк/процессинг | **Оплата не проходит** (банк / процессинг / ДУЭТ) |
| Вопросы по отчётам + не печатается отчёт (+ сверки) | **Отчёты и расхождения данных** |
| Карта клиента + лимиты + «клиенты» из Другое | **Карты питания и клиенты** |
| Отменить чек | **Корректировка чека** (сторно, возврат, удаление, перенос) |
| Консультация | **Консультация / звонок** |
| Не работает ЧЗ | **ЧЗ / маркировка** (оперативный инцидент) ≠ расследование отклонений |

### 4.2 Catch-all и дыры

| Было | Станет |
|------|--------|
| «Другое» (~37% всей выборки; без MFC дыра жива) | Убрать после 1–2 нед параллели; остаток → **Нестандарт + Суть** |
| «Доп. настройки, сопутствующий сервис…» (широкая свалка) | Сузить до **настроек и доступов**; раскатки/запуски/сервисы — отдельные пункты |
| НДС, раскатки, запуски, банк-модуль в catch-all | Явные пункты (ниже) |

### 4.3 Новые / явные пункты (раскатки и хвост)

| Станет (новый typical) | База | Объём | Комментарий |
|------------------------|------|-------|-------------|
| Массовое обновление / раскатка ПО | **T5** | × N | Сейчас часто «на глаз» 100–1000 |
| Массовое обновление НДС / фиск. параметров | **T5** | × N | Кампания конфига, не точечный инцидент |
| НДС: точечная ошибка ставки / режима | T5 / C15 | обычно 1 | Ручной хвост после авто |
| Банк / модуль банка (ручная доводка) | **C15** | × N при пакете | Не путать с «оплата не проходит» |
| Запуск / подготовка объекта (или партии) | S30 или C15 | × N | **Одно** имя; не плодить «Оборудование / запуск» |
| Автообновление: сбой / офлайн / доводка | T5 / C15 | × N | После агента — **это** в премии |
| Сопутствующий сервис клиента | C15 | 1 | Work Bright / AllBusiness… |
| Настройки и доступы (не блокирует) | T5 | 1 | Остаток бывших «Доп. настроек» |

### 4.4 Автообновление (продуктовый эффект)

| Было | Станет |
|------|--------|
| Ручные массовые раскатки ПО в премии инженера (~15% фонда баллов по оценке) | Цель: агент делает успешную раскатку; **вне** премии инженера |
| Фейлы/офлайн смешаны с общей раскаткой | Отдельный typical «сбой / офлайн / доводка» × N |

## 5. Способы решения (`solution_method`)

| Было | Станет |
|------|--------|
| «Другое» — доминирующий выбор | Убрать после параллели; дыры закрыть явными пунктами |
| Рестарт ПО + отдельно GriServ | **Рестарт ПО / сервиса** (в т.ч. GriServ) |
| Провода + питание | **Железо: питание / кабели / переподключение** |
| Обмен + SALES3 | **Обмен / перевыгрузка данных** |
| «Проверить бумагу» (мёртвый) | Убрать |
| «Настройка ПО и доступы» — слишком широко | **Доступ / код кассира** + **точечная настройка конфига** |
| Нет явных: консультация, раскатка, карты, НДС, доводка авто | Добавить (см. [13](13-solution-catalog-proposal.md)) |

Катить **вместе** с ревизией typical.

## 6. MFC и контуры данных

| | Было | Станет |
|---|------|--------|
| MFC-перенос (~33% заявок) | Typical часто пусто / «Другое»; техзакрытие | KPI каталога — **без MFC-bulk** |
| Баллы за техbulk / компенсатор | Риск накрутки или дыры; дневной компенсатор | **Без компенсатора.** Убрать пустые техзаявки; массовый перенос — форма с выбором **typical + solution** на каждую заявку → обычная норма |
| Смешение контуров в одном рейтинге | Одна куча | Натив / MFC / почта·чаты — разные контуры учёта (по мере очистки переноса) |

## 8. Выезды и дежурство

| | Было | Станет |
|---|------|--------|
| Баллы выезда | Мск 30 / ближнее 60 / дальнее 120; иногда «крупная» сумма | **Только 60** при `departure` |
| Typical / осложнение при выезде | Могли суммироваться с выездом | **Игнорируются** в формуле баллов |
| Дежурство ×2 | Чекбокс `watch`, риск рассинхрона | ×2 к 60 → 120 по **табелю**; две статьи в отчёте |

---

## 9. Что не меняем в фазе 0–1

- Курс **15 ₽ / балл**
- Саму идею баллов и премии за закрытые заявки
- Коэффициент дежурства ×2
- Текущие выплаты до даты X (фаза 1–2 = каталоги + тень)

---

## 10. Порядок внедрения (после утверждения этого документа)

```
Фаза 0  → утвердить «было/станет» (этот файл + docs/18)
Фаза 1  → поля и каталоги в Okdesk (выплаты ещё старые)
Фаза 2  → теневой расчёт 4–6 нед
Фаза 3  → дата X: платим по формуле
Фаза 4+ → сегменты, расследования, автообновление, аудит
```
