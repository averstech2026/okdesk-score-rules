# Потоки работ: конвейер vs общий пул

## Проблема

Одна абсолютная сумма баллов смешивает разные виды труда:

- **Конвейер** — большие контракты, много типовых повторяющихся заявок, высокий штучный объём (пример: ~50 заявок/день).
- **Общий пул** — разнородные клиенты, больше времени на контекст и разбор, меньше штук при сопоставимой нагрузке.

При премии «кто набрал больше баллов» рациональная стратегия — закрепиться на конвейере. Это баг метрики.

## Сегментация

Каждая заявка (или исполнитель в периоде) относится к сегменту:

| Сегмент | Как определять в Okdesk (нужно зафиксировать) |
|---------|-----------------------------------------------|
| `conveyor` | Клиент / договор / очередь / тег контрактного потока |
| `pool` | Общая очередь / остальные клиенты |
| `field` | Опционально: преимущественно выездная работа за период |

Точный список конвейерных клиентов/очередей — в [open-questions.md](open-questions.md).

## Как считать вклад

Минимум (v1):

1. Считать и показывать баллы **по сегментам отдельно**.
2. Не строить «чемпион месяца» только по сумме всех баллов без контекста сегмента.
3. Сравнивать человека с **медианой своего сегмента** (или с нормой сегмента), а не только с абсолютом коллеги с другого потока.

Возможно позже (v2):

- нормативные ожидания: на конвейере — выше штук и ниже средний вес; в пуле — наоборот;
- коэффициент микса, если человек смешанно работает в обоих сегментах.

## Управление потоками

- Работа в пуле не должна быть «наказанием» относительно конвейера.
- Баланс людей на пуле — зона руководителя (и владельца очереди), а не только добровольный выбор удобного потока.
