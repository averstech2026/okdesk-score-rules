## Формула

```
если выезд (departure):
  баллы = 60 × (1|2 дежурство)
иначе:
  баллы = (база × N + осложнение*) × (1|2)
* осложнение только если уровень И непустое описание
```

| Код | Баллы | Смысл |
|-----|------:|-------|
| T5 | 5 | типовой инцидент |
| C15 | 15 | Нестандарт (тип заявки) |
| S30 | 30 | разбор / статья БЗ |
| H60 | 60 | оборудование / запуск |

Выезд всегда **только 60**, без осложнений.

**Префиксы `[5]` / `[30]` / `[60]` в именах Okdesk пока не ставим** — чтобы не тянуть выбор к «дорогим» пунктам. Несколько недель / месяц выбирают по смыслу проблемы; база живёт в этой таблице и в скриптах. Потом решим, возвращать ли подсказки в UI.

### Не заводим / вне списка

| Было в черновике | Решение |
|------------------|---------|
| Сопутствующий сервис (Work Bright / AllBusiness…) | **Убрать** → тип **Нестандарт** + в Сути что за сервис |
| Отдельные «Доступ / код» и «Настройки и доступы» | **Слиты** в №12 |
| Отдельные «Оборудование» и «Запуск / подготовка» | **Слиты** в №22 |
| «Другое» | не заводить |
| Автообновление: сбой… | не заводить → №18 |

Тип заявки **Нестандарт** + «Суть» → база **15**. На этом типе поле `typical` **не показываем** / не обязателен (иначе спотыкаются: «Другое» убрали, выбирать нечего). Способ решения — из каталога.

**Цикл доработки каталога:** первые недели смотрим тексты в «Суть» (в т.ч. Work Bright / AllBusiness и разовое). Если класс повторяется — добавляем пункт в typical; пока не плодим справочник наугад.

## Не смешивать

| Этот | Не путать с |
|------|-------------|
| ЧЗ оперативный (10) | Расследование (20) |
| Оплата не проходит (6) | Банк / модуль (17) |
| Расхождение смены (14) | Расследование (20) |
| НДС точечный (16) | Обновление НДС (19) |
| Обновление / раскатка ПО (18) | Подключение / запуск (22) |
| Статья БЗ (21) | Консультация (13) |
| Периферия (8) | Не работает касса (5) |
