# Баланс потоков: конвейер vs расследования

## В чём специфика

У поддержки не один вид труда:

| Поток | Примеры | Характер |
|-------|---------|----------|
| **Конвейер** | Типовые инциденты больших контрактов (IFCM, MFC и др.), повторяющиеся сценарии | Много штук, низкая неопределённость, быстрый цикл |
| **Сложный разбор** | Анализ расхождений, нарушения/отклонения ЧЗ, транзакционные расследования, сверки «почему не бьётся» | Меньше штук, больше времени, экспертиза, часто эскалация к целевому сотруднику |

Одна сумма баллов без правил маршрутизации ломает оба конца:

- специалист по расследованиям проигрывает «стахановцу» на конвейере;
- человек с конвейера может **перехватить** «жирные» кейсы расхождений/ЧЗ, закрыть пачкой у себя и не отдать целевому.

## Цель баланса

1. Платить за реальный тип труда (объём ≠ расследование).
2. Сложные кейсы должны доходить до **целевого** исполнителя.
3. Перехват и штамповка расследований на конвейере — детектируемы и невыгодны.

## Как учесть в планируемой реализации

### 1. Явные категории в каталоге проблем

Выделить (имена уточнить с тимлидами), например:

- **Расхождение данных / сверка** (уже в целевом каталоге);
- **Расследование / отклонение ЧЗ** (письма RE:, МРЦ, просрочка, прецеденты) — не смешивать с «Не работает ЧЗ» (оперативный инцидент);
- **Транзакционное / финансовое расследование** (если отличается от обычной сверки отчётов).

У оперативного «Не работает ЧЗ / не печатает» и у **расследования отклонений ЧЗ** — разные пункты и разные нормы баллов.

### 2. Два трека учёта (сегменты)

Как уже в принципах «конвейер vs пул», но явнее:

| Трек | Кто | Метрика премии |
|------|-----|----------------|
| A. Объём / конвейер | Закреплённые на контракте | Норматив штук/баллов **своего** сегмента |
| B. Разбор / расследования | Целевые сотрудники (список) | Норматив по сложным категориям + качество |

Смешанный сотрудник: вклад считается **по трекам отдельно**, не одной кашей.

### 3. Маршрутизация важнее «честности на честном слове»

В Okdesk (без Expert — то, что можно кликами):

- отдельная **группа / очередь** «Расследования / сверки / ЧЗ-контроль»;
- заявки с typical ∈ {расхождение, расследование ЧЗ, …} → по таблице маршрутизации на эту группу / целевого;
- у роли конвейера — **нет права** (или нет привычки в процессе) забирать из этой очереди; либо право есть, но закрытие «мимо» целевого режется аудитом.

Если маршрутизация по доп. атрибуту ограничена тарифом — минимум: **регламент** «такие typical только на Иванова/группу X» + еженедельный отчёт «кто закрыл расследования».

### 4. Баллы: высокая база, но только «легально»

- Категории расследования — **выше** базы, чем типовой инцидент (иначе специалист некормлен).
- Но начисление полного веса расследования — если:
  - исполнитель ∈ список целевых **или**
  - была явная передача (смена ответственного на целевого) **или**
  - тип/очередь = расследования.

Иначе (конвейерный исполнитель закрыл «расследование» сам):

- либо **норматив конвейера** (низкий вес, как за обычную заявку того клиента);
- либо заявка не идёт в премию как расследование до разбора;
- плюс флаг на аудит.

Так перехват перестаёт быть выгодным: пачкой «штамповать расследования» ради баллов бессмысленно.

### 5. Анти-паттерн «штампую расхождения на конвейере»

Сигналы для отчёта/аудита:

| Флаг | Смысл |
|------|--------|
| Исполнитель из конвейерного сегмента + typical = расследование/расхождение | Должен был передать? |
| Много таких закрытий за период у одного человека | Системный перехват |
| Короткое время + высокий балл + категория расследования | Штамп, не разбор |
| Автор/первый ответственный ≠ целевой, закрыл сам без передачи | Обход маршрута |
| Одинаковые формулировки пачкой | Копипаст «расследований» |

Разбор флагов — у тимлида; повтор → корректировка премии / эскалация.

### 6. Регламент передачи (одна страница для команды)

1. Увидел расхождение / расследование ЧЗ / транзакционный разбор → **не закрывать у себя**, если ты не целевой.
2. Выставить typical из блока «разбор», назначить / передать в группу расследований.
3. Целевой ведёт до решения; при необходимости конвейерный остаётся соисполнителем (если так принято).
4. Исключения (целевой в отпуске и т.д.) — только с пометкой и согласованием.

## Связь с уже согласованным

| Уже есть | Как стыкуется |
|----------|----------------|
| Сегменты конвейер vs пул | Добавить подсегмент **«расследования»** или отдельный трек B |
| Каталог typical | Развести оперативный ЧЗ и расследование отклонений ЧЗ |
| Тип «Нестандарт» | Не для массовых расхождений — для них отдельные пункты каталога; Нестандарт — редкий хвост |
| Способы решения | Для расследований чаще: сверка/БД/обмен/сеть, не «консультация» |
| Аудит | Флаги перехвата — обязательная часть инструмента |

---

## Чего не делать

- Не поднимать всем баллы «за сложность» без маршрута — конвейер начнёт клеить сложные typical.
- Не оставлять только регламент без отчёта — через месяц забудут.
- Не мешать MFC-перенос с настоящими расследованиями в одной статистике.

---

## Минимальный MVP баланса

1. 2–3 явных typical для разбора (расхождение; расследование ЧЗ; при необходимости транзакции).  
2. Список целевых + правило «передавай».  
3. В премии: сложные категории считают полный вес только у целевых (или после передачи).  
4. Еженедельный отчёт: кто из конвейера закрыл сложные категории.

Дальше — очередь в Okdesk и автоматические флаги в `scripts/analyze_scores.py`.
