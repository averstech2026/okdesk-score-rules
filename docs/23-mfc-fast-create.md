## 1. Проблема «как сейчас»

1. **Петров Артём** в течение дня получает обращения из разных источников:
   - внешний ServiceDesk MFC/UCG (`help.ucg.ru`);
   - чаты;
   - звонки на поддержку.
2. Полноценно оформлять каждую карточку в Okdesk **нет времени** → пишет **текстовый список** (часто через Надежду в чат).
3. **Королева Надежда** вручную создаёт заявки в Okdesk: компания MFC, ответственный Артём.
4. Отдельным скриптом заявки **технически закрывают** (тип/способ часто «Другое»).

Итог в данных (~6 мес): компания **ООО эМэФСи (МФС, MFC)** = `company_id` **9**, ~33% выборки; у Артёма (~id **5**) основная масса; автор часто Надежда (id **13**); описания пустые; `typical`/`solution_method` «Другое» или пусто; объект обслуживания (`service_object_id`) у MFC часто уже бывает, но при массовом переносе качество заполнения слабое.

Менеджера из цепочки **убираем** — оформляет **Артём**.

## 2. Решение: вариант A — веб-страница

Захостить у нас (ресурс для хостинга уже есть).

### Кто и что делает

1. Артём **вставляет общий список** (как сейчас в чате/файле).
2. Страница режет на строки → таблица.
3. По **каждой** строке обязательно:
   - **объект обслуживания MFC** (у компании много объектов — поиск, не длинный scroll);
   - **типовая проблема** (`typical`);
   - **способ решения** (`solution_method`).
4. Одна кнопка: **создать и сразу закрыть** пачкой.
5. Комментарий при закрытии: пометка **«Техническое решение»** + способ из строки (+ опционально общий текст на пачку).
6. На экране — ID созданных заявок / ошибки по строкам.

### Пресеты карточки Okdesk

| Поле | Значение |
|------|----------|
| Компания | MFC, `company_id = 9` |
| Ответственный | Артём, `assignee_id = 5` |
| Тип заявки | `service` (Обслуживание) |
| Автор | Артём (API-токен сотрудника) |
| Объект | `maintenance_entity_id` = выбранный объект строки |
| `typical` | выбранный |
| `solution_method` | выбранный |
| `ticket_weight` | **не ставить** (сегмент MFC вне общей премиальной формулы — см. контуры / bonus) |

На день по MFC обычно есть **сводная («компенсирующая») заявка** вроде «МФС. Заявки, звонки» с крупным баллом за объём дня. Техзакрытия из этой формы **без баллов — норма**: баллы несёт компенсатор, не каждая мелкая карточка.

### API (Okdesk)

- `POST /api/v1/issues` — создание  
- `POST /api/v1/issues/{id}/statuses` — переход в `completed` или `closed` + comment  
- Токен **только на бэкенде** (прокси); браузер токен не видит.

Статус техзакрытия (`completed` vs сразу `closed`) — **открытый вопрос**, уточнить по текущему скрипту/регламенту.

## 3. Формат входного списка (реальный пример)

Список приходит **смешанный**: ссылки ServiceDesk и короткие заголовки вперемешку. Пример (фрагмент от Надежды, ~4 дня пачками):

```text
https://help.ucg.ru/Task/View/136419
https://help.ucg.ru/Task/View/136416
Сбер. Оплата
https://help.ucg.ru/Task/View/136425
НК НПЗ. Чек
ПП12. Закрыть смену
ОМК. Клиенты
Маг 15\1. Закрыть смену
```

### Правила парсера

| Правило | Деталь |
|---------|--------|
| Одна непустая строка = одна заявка | Пустые строки отбрасывать |
| URL `help.ucg.ru/Task/View/{id}` (IntraService) | `source_type=url`, `external_id`, `external_url`; title пока `UCG #{id}` |
| То же `/Task/Edit/{id}`, query, `<url>`, markdown `[…](url)` | тоже распознаётся |
| Подтянуть Name/Description из SD | только с Basic auth (`INTRASERVICE_*` в env) → галка enrich в UI |
| Иначе | `source_type=text`, `title` = строка как есть |
| Строка между URL — **не** заголовок группы | `Сбер. Оплата` = отдельная заявка |
| Не требовать новый формат от людей | Брать как пишут сейчас |
| Объём | Ориентир 40–70 строк/день, пачка за раз |

### Паттерн текста: `Объект/клиент. Суть`

Примеры: `ПП12. Закрыть смену`, `НК НПЗ. Чек`, `Аскона. Не печатаются чеки`.

После первой `.`:

- `object_hint` → fuzzy-подсказка объекта MFC  
- `typical_hint` → fuzzy-подсказка типовой проблемы  

Нормализация hint: `пп12`→`ПП12`, `Нк НПЗ`→`НК НПЗ`, `Маг 15\1` ≈ варианты написания.

Для **чистых URL** hint нет → объект и typical только руками (ссылка кликабельна, чтобы глянуть карточку в UCG).

## 4. Модель строки в UI

```text
source_type:    url | text
title:          текст или "UCG #136419"
external_url:   https://help.ucg.ru/Task/View/… | null
external_id:    136419 | null
object_id:      обязателен
object_hint:    из префикса (если text)
typical:        обязателен
typical_hint:   из хвоста (если text)
solution:       обязателен
object_count:   опционально (N) — код `object_count`
complication_level: опционально `+15` | `+30`
complication:   описание осложнения (обязательно вместе с уровнем)
description:    описание заявки (при N>1 — список объектов)
selected:       галка (можно снять мусор)
```

Поля N / осложнение / описание осложнения — из [21-okdesk-new-fields.md](21-okdesk-new-fields.md).

В Okdesk дополнительно:

- в description или в комментарии закрытия — **ссылка на UCG**, если была;
- custom_parameters: `typical`, `solution_method`, при заполнении — `object_count`, `complication_level`, `complication`;
- create → close + комментарий вида:

```text
Техническое решение.
Способ: {solution_method}
{опциональный общий текст пачки}
```

### UX

- Вставка списка → «Разобрать» → таблица.  
- Колонки: галка | тема | объект | typical | solution | **N** | **осложнение** | **описание осложнения** | **описание** | UCG.  
- Списки typical/solution — **укороченные** (топ конвейера MFC), не весь справочник Okdesk.  
- Объекты — только компании `9`, с поиском.  
- Массово: проставить solution и N отмеченным; фильтр «без объекта».  
- Кнопка блокирует, если нет объекта/typical/solution или осложнение без пары уровень+текст.

## 5. Архитектура

```text
[статическая страница] → [наш бэкенд-прокси] → avers.okdesk.ru
                              ↑
                    OKDESK_API_TOKEN (env)
```

Эндпоинты бэкенда (черновик):

- `GET /api/mfc/objects?q=` — объекты компании 9  
- `GET /api/mfc/catalogs` — укороченные typical + solution  
- `POST /api/mfc/batch` — массив строк → create + close, ответ с id/ошибками  

Парсинг списка можно делать на клиенте.

Стек: Python **FastAPI** + статическая страница. Код: [`tools/mfc-fast-create/`](../tools/mfc-fast-create/) (см. README там).

## 6. Что сознательно не делаем в MVP

- Интеграция «забрать все тикеты из help.ucg.ru автоматически» (долгий горизонт).  
- Оформление через полный UI Okdesk / клонирование шаблона как основной путь.  
- Прогон MFC через тип «Нестандарт» (`docs/12`).  
- Начисление баллов из общей премиальной формулы на этот bulk.  
- Замена роли Артёма ботом в Telegram (вариант C — следующий этап, если страница приживётся).

## 9. Контекст разговора (кратко)

| Тема | Договорённость |
|------|----------------|
| Вариант | **A** — веб-форма, не файл-скрипт как основной UX и не Telegram в MVP |
| Исполнитель | **Артём**, менеджер (Надежда) из цепочки убран |
| Объект | **Обязателен** на каждую строку (у MFC много объектов) |
| Typical + solution | Обязательны; не дефолтить «Другое» |
| Закрытие | Сразу create+close одной кнопкой |
| Комментарий | «Техническое решение» + способ (+ опционально общий текст) |
| Вход | Реальный формат чата: **URL UCG + короткие заголовки** |
| Автоподсказки | Из `Префикс. Суть` → объект и typical |
| Хостинг | Свой ресурс уже есть; токен только на сервере |
| Репо score-rules | Фиксация ТЗ здесь; реализация — следующий сеанс |
