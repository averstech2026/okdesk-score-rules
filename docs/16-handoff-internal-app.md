# Передача в соседний проект: личные отчёты, премия, дежурства

Документ для **внутреннего приложения** (таймлайн, смены, отпуска).  
Работа по правилам баллов и ревизии каталога остаётся в репозитории [okdesk-score-rules](https://github.com/averstech2026/okdesk-score-rules). Здесь — только то, что нужно **реализовать в другом проекте**.

**Цель фичи:** убрать ручную пересылку отчётов и автоматизировать / структурировать начисление по заявкам на основании выполненной работы.

---

## 1. Контекст «как сейчас»

### 1.1 Основная премия (месяц)

1. Руководитель ежемесячно выгружает отчёты из Okdesk.
2. Суммирует баллы по инженеру.
3. Умножает на **15 ₽ / балл** → сумма премии.

Инженер сейчас часто сам ставит баллы «на глаз». Целевая модель (см. `docs/00–07` в этом репо): баллы считает **правило** по категории / типу, а не свободный ввод.

### 1.2 Дежурства (отдельный цикл ~1,5 недели)

1. Инженеры **сами** формируют отчёт и присылают руководителю.
2. В отчёте уже посчитана сумма. Она включает:
   - баллы заявок, закрытых в дежурную смену, с коэффициентом **×2**;
   - плюс **ставки по дням смены**: будни / выходные и праздники (это отдельно от баллов ×2).

Источник истины для «это дежурство» — **график/табель смен** (в соседнем проекте он уже есть), а не галочка инженера.

### 1.3 Данные по заявкам (уже есть в соседнем проекте)

- Заявки Okdesk тянутся **накопительно** в **Firestore**.
- Google Apps Script раз в ~**2 часа** обогащает базу из Okdesk.
- Каждый прогон тянет **только дельту** с прошлого раза (лимиты бесплатного тарифа Firestore).

Для отчётов премии/дежурств опираться на эту базу, а не на разовые ручные выгрузки.

## 2. Что сделать в соседнем проекте

### 2.1 Личный раздел сотрудника

У каждого сотрудника — свой кабинет с **автоматически сформированными** отчётами по заявкам Okdesk + правилам.

| Тип отчёта | Период | Содержание |
|------------|--------|------------|
| **Премия** | календарный месяц | Список/сводка заявок → баллы по правилам → сумма **баллы × 15 ₽** |
| **Дежурство** | цикл дежурной смены (~1,5 нед. / по табелю) | Заявки в окне смен → баллы ×2 + **дневные ставки** (будни / вых·праздн.) → итоговая сумма |

**В каждом отчёте обязательно:**

1. **Прозрачные правила** — кратко, как посчитано (формула, курс балла, ×2, ставки дней, что вошло / не вошло).
2. **Детализация по заявкам** (id Okdesk, дата, тема/тип, начисленные баллы, признаки: выезд, дежурство и т.д.).
3. **SLA по сотруднику за период** — пока **справочно**, без влияния на сумму. Дальше решим, вводить ли в формулу.
4. Статус отчёта: черновик / на проверке / **принят** / (при необходимости) отклонён с комментарием.

**Позже (не блокер MVP):** в списке заявок отчёта показывать **типовые баллы по типу заявки** — когда утвердим новый подход к каталогу (`docs/01`, `docs/10`).

### 2.2 Массовое создание / закрытие заявок (контур MFC и аналоги)

В том же личном разделе — инструмент **массовой обработки** заявок (не только для одного инженера):

- сотрудник готовит список (из своих записей / импорта);
- выбирает **типы** и **способы решения**;
- массово создаёт / закрывает заявки в Okdesk (или через уже существующую интеграцию).

Контекст процесса MFC (важно не смешивать с «живой» классификацией):

- у MFC свой helpdesk; заявки часто переносят скопом **только по заголовку**;
- техзакрытия без баллов — норма; баллы часто несёт **сводная** заявка дня («МФС. Заявки, звонки» и аналоги);
- MFC — **отдельный сегмент** конвейера, не смешивать с общим пулом при KPI/аномалиях.

Подробнее: `docs/11-ticket-contours.md` в этом репо.

### 2.3 Раздел руководителя (и зама)

Роли: **руководитель** + **зам** (явный доступ).

| Возможность | Описание |
|-------------|----------|
| Все отчёты сотрудников | Премия и дежурства за выбранный период |
| Просмотр | Детализация, правила расчёта, SLA-справка |
| Подтверждение | Смена статуса на **«отчёт принят»** (и откат/комментарий при правках) |

Без этого шага автоматизация неполная: сейчас принятие — через переписку/файлы.

## 3. Правила начисления (кратко, для реализации)

Полные тексты — в этом репозитории. Ниже — контракт для кода отчётов.

### 3.1 Слои баллов заявки

```
итоговые_баллы ≈
  если выезд: 60 × коэффициент_дежурства
  иначе: (база_категории × N + осложнение 0|15|30) × коэффициент_дежурства
```

| Слой | Суть | Источник правил |
|------|------|-----------------|
| База | Категория работ (ориентиры: T5=5, C15=15, S30=30) | `docs/01-base-catalog.md` |
| Осложнения | Поле-**список** Okdesk: +15 или +30 + текст | `docs/02-complications.md` |
| Выезд | Чекбокс → **только 60** (без суммы с typical) | `docs/03-field-visits.md` |
| Дежурство ×2 | Только если заявка в окне **своей** дежурной смены | `docs/04-duty.md` + табель соседнего приложения |

Пакет по объектам (раскатка / массовое обновление): `база × N_объектов`, N явен в заявке.

### 3.2 Премия

```
сумма_премии_₽ = сумма_нормативных_баллов_за_месяц × 15
```

- Курс **1 балл = 15 ₽** — действующая практика (зафиксировать константой, легко менять).
- Групповой SLA **не** штрафует личную премию. В отчёт — персональный SLA после взятия, пока справочно (`docs/06-sla.md`, `docs/07-bonus.md`).
- Не смешивать в одной сумме без сегментов: конвейер (MFC/IFCM…) vs общий пул (`docs/05-streams.md`).

### 3.3 Дежурство: баллы ×2 **и** дневные ставки

Два независимых слагаемых:

1. **Балльная часть:** заявки, попавшие в дежурную смену → `(база + …) × 2` → × 15 ₽ (или как принято сейчас в дежурных отчётах — уточнить при сверке с образцом старого отчёта).
2. **Дневные ставки смены** (из табеля):
   - ставка **будний** день дежурства;
   - ставка **выходной / праздничный** день дежурства.

Конкретные суммы ставок дней — **взять из текущих шаблонов отчётов инженеров** / внутренних нормативов (в `okdesk-score-rules` числовых ставок дней нет — только ×2 по баллам).

Вне своей дежурной смены (ночная раскатка и т.п.) **отдельного** коэффициента пока нет.

### 3.4 Двойной учёт

Заявки дежурного цикла **не должны** дважды раздувать месячную премию, если так принято сейчас.  
Правило стыковки (дежурный отчёт vs месяц) — **явно зафиксировать** при внедрении (см. open-questions в этом репо).

## 4. Статусы и жизненный цикл отчёта (предложение MVP)

```
auto_draft  →  submitted (опционально)  →  accepted
                  ↘ rejected / returned
```

| Статус | Кто | Смысл |
|--------|-----|--------|
| `auto_draft` | система | Пересчёт по заявкам + табелю; сотрудник видит |
| `submitted` | сотрудник (опц.) | «Готово к проверке» |
| `accepted` | руководитель / зам | Отчёт принят к начислению |
| `returned` | руководитель / зам | Нужна правка / комментарий |

Пересчёт черновика возможен, пока не `accepted` (политика: после принятия — только ручная корректировка админом).

## 5. Зависимости между проектами

| Что | Где живёт |
|-----|-----------|
| Нормы баллов, каталоги, контуры MFC, SLA-принципы | `okdesk-score-rules` (документы + анализ) |
| Табель смен, отпуска, календарь дежурств | соседнее приложение |
| Копия/индекс заявок Okdesk | Firestore соседнего приложения (+ GAS дельта) |
| UI отчётов, массовые операции, accept | **соседнее приложение** |
| Курс 15 ₽, дневные ставки дежурства | конфиг соседнего приложения (числа из текущей практики) |

Рекомендация: правила расчёта баллов держать **версионируемым конфигом** (или ссылкой на утверждённый snapshot из `okdesk-score-rules`), чтобы в отчёте было видно «по правилам vX».

## 6. MVP vs позже

### MVP

- [ ] Личный раздел: отчёт **премия** (месяц) из Firestore-заявок + правила + ×15 ₽.
- [ ] Личный раздел: отчёт **дежурство** (смены из табеля) = баллы ×2 + ставки будни/вых·праздн.
- [ ] Текст правил внутри отчёта.
- [ ] SLA сотрудника за период — справочный блок.
- [ ] Админ-раздел: все отчёты + статус «принят» (руководитель + зам).
- [ ] Массовое создание/закрытие заявок (типы + способы) для инженеров конвейера/MFC.

### Позже

- [ ] Типовые баллы по типу заявки в строках отчёта (после утверждения нового каталога).
- [ ] Качество / аудит / сегменты конвейер vs пул в формуле премии.
- [ ] Учёт автообновлений ПО (массовые раскатки уходят из ручной премии) — см. `analysis/mass-update-auto.html`.

## 7. Открытые вопросы для соседнего проекта

Заполнить до или в начале разработки:

1. Точные **дневные ставки** дежурства (будни / выходные·праздники) и входит ли ×15 только в балльную часть.
2. Граница периода дежурного отчёта: строго по табелю или фиксированные окна ~1,5 недели.
3. Как матчить заявку к смене: по `completed_at` / `closed_at` / времени назначения.
4. Нужен ли шаг `submitted` от сотрудника или достаточно авто-черновика + accept руководителем.
5. API Okdesk для массового create/update: лимиты, поля typical/solution, ответственный.
6. Список сотрудников с доступом admin (руководитель + зам) — идентификаторы в auth.
7. Политика пересчёта после `accepted`.
8. Двойной учёт: дежурные баллы внутри/вне месячной премии.
