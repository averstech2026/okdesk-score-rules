План действий (согласованная логика)

Порядок важен: сначала каталоги и «Нестандарт» (без Expert), потом нормы баллов и сегменты, потом аналитика/аудит. Полный пошаговый план внедрения (фазы 0–6, тень, дата X, риски): docs/17-implementation-plan.md.

Фазы внедрения (кратко) 0 подготовка → 1 каталоги в Okdesk → 2 теневой расчёт (4–6 нед) → 3 нормы и премия с даты X → 4 потоки/расследования → 5 автообновление → 6 аудит. Выплаты меняются только с фазы 3.
  1. Зафиксировать контуры данных
    Нативные заявки / перенос MFC (~33%, техзакрытие, typical часто пусто) / почта·чаты. KPI качества заполнения и каталога — без MFC. Документ: docs/11-ticket-contours.md.
  2. Обновить справочник типовых проблем
    Слить дубли (ДУЭТ+банк, отчёты, карта+лимиты→«Карты и клиенты»). Добавить блок раскатки и ручной хвост (см. ниже и docs/10): массовая раскатка ПО / массовый НДС-конфиг (база × N); точечный НДС; банк-модуль вручную; запуск/подготовка; фейл/офлайн автообновления. Сузить «Доп. настройки…». Убрать «Другое». Связь с автообновлением: analysis/mass-update-auto.html.
  3. Ввести тип заявки «Нестандарт» + поле «Суть»
    Без тарифа Expert 50+: автоправила «чекбокс→обязательное поле» недоступны. Тип «Нестандарт» = нестандартная проблема; способ решения всё равно из каталога. Обязательность «Сути» при «Решена». MFC этим типом не кормить. docs/12-nonstandard-issue-type.md.
  4. Обновить справочник способов решения (~14–15)
    Слить рестарты, железо, обмены; убрать «бумагу»; разрезать «Настройка ПО». Добавить: консультация, раскатка, карты/начисления, доступ/код кассира, НДС. Убрать «Другое» как основной путь. Катить вместе с каталогом проблем. docs/13-solution-catalog-proposal.md.
  5. Нормы баллов: категория → база (+ надбавки), не свободный select 5…1000
    Сейчас ticket_weight — длинный выпадающий список чисел. Цель: баллы из правил. Без выезда: база + осложнение +15/+30. С выездом: только 60. Дежурство ×2 по табелю.
  6. Сегменты премии: конвейер ≠ общий пул
    IFCM, MFC и т.п. — отдельные нормы/сравнение с медианой сегмента. Не один рейтинг «кто набрал больше баллов».
  7. Трек расследований + защита от перехвата
    Отдельные typical для расхождений / расследований ЧЗ / транзакций. Маршрут на целевых. Полный «сложный» вес — только целевым или после передачи; иначе конвейерный вес + флаги аудита. См. § «Конвейер vs расследования».
  8. SLA: группа ≠ человек
    Просрочка очереди — термометр + владелец пула/смены. Лично — сроки после взятия. SLA как лёгкое дополнение к баллам, не вместо.
  9. Аналитика / аудит (инструмент)
    Скрипты уже есть: scripts/fetch_issues.py, scripts/analyze_scores.py. Дальше: флаги аномалий, доля Нестандарт без MFC, микс категорий, хвост высоких баллов. Внеурочка вне дежурства — пока не трогаем.
Ближайший практический шаг в Okdesk (можно без Expert) 1) Черновик новых списков typical + solution утвердить с тимлидами. 2) Создать тип «Нестандарт» + атрибут «Суть». 3) Параллельно 1–2 недели, затем убрать «Другое» из обоих списков. 4) Через месяц — разбор формулировок «Суть» → донастройка каталога проблем.

Контекст: как работали и что болело

Как сейчас
  • Инженер сам ставит баллы (select 5…1000), ориентиры «5 / 15 / 30» условные
  • Дежурство: чекбокс watch, множитель ×2 (сейчас вручную)
  • Выезд: чекбокс → только 60 (typical не суммировать)
  • Осложнение (план): поле-список Okdesk +15 / +30 + текст причины
  • Обязательные: типовая проблема, способ решения, тип кассы…
  • Премия по сумме баллов (месяц) + отдельно дежурства (~1.5 нед.)
Изъяны
  • Нет чёткой градации веса; разночтения между инженерами
  • Самооценка → риск приписок (~1000+ заявок/мес, вручную не проверить)
  • Один ПИОТ может занять полдня при тех же 30 баллах → выгоднее 6 простых
  • Конвейер 50 типовых vs общий пул — одна сумма баллов врёт
  • «Другое» в проблеме и особенно в способе — свалка

Принципы (договорённости)

  • Сохраняем баллы, ×2 дежурства, фикс выездов по зоне
  • Убираем свободный ввод баллов → категория/правило (+ закрытые надбавки за осложнение)
  • Конвейер и пул не мешать в одном «чемпионе месяца»
  • SLA группы ≠ персональный минус; лично — после взятия
  • Внеурочка вне смены — пока без отдельного коэффициента
  • Нестандарт = тип проблемы; способ — из списка
  • Тариф ниже Expert 50+: нет автоправил от поддержки Okdesk → тип «Нестандарт» вместо условных полей

Формула (цель): (база + осложнения + выезд) × (1|2 дежурство). Подробнее: docs/00-principles.md … 07-bonus.md.

Тип «Нестандарт»

Роль Нестандарт = нестандартная проблема (нет пункта в typical). Способ решения при этом выбирается из каталога (рестарт, обмен, консультация…). Нестандартная заявка ≠ «неизвестно что сделали».
В Okdesk
  • Тип nonstandard / «Нестандарт»
  • Атрибут nonstandard_detail «Суть нестандарта»
  • Показать атрибут только на этом типе
  • Обязателен при входе в «Решена»
  • Убрать «Другое» из typical
  • MFC-bulk — не через этот тип
Почему не чекбокс

Идея «чекбокс → обязательное поле» хорошая, но без Expert 50+ поддержка Okdesk автоправила не делает, а условная обязательность атрибутов в UI не настраивается.

Тип заявки даёт обязательность «только когда нужно» штатными средствами.

Баланс: конвейер vs расследования

Кроме «конвейер / общий пул» есть трек сложного разбора: расхождения, нарушения/отклонения ЧЗ, транзакционные расследования. Их нельзя кормить той же логикой, что 50 типовых в день — и нельзя давать конвейеру штамповать их пачками вместо передачи целевому.

Риск перехвата Сотрудник конвейера ловит кейс «расхождение / расследование ЧЗ», закрывает у себя пачкой ради баллов и не отдаёт целевому специалисту. Если за это платить полным «сложным» весом — система поощряет перехват.
Как учесть в модели
  • Отдельные пункты typical: оперативный ЧЗ ≠ расследование отклонений ЧЗ; плюс расхождения / транзакционный разбор
  • Трек A (объём/конвейер) и трек B (разбор) — учёт и премия по трекам
  • Маршрут: сложные typical → группа/целевой; регламент «не закрывай у себя, передай»
  • Полный вес расследования — только у целевых или после передачи; иначе вес как у конвейера / аудит
Анти-штамп (флаги)
  • Конвейерный исполнитель + typical «расследование/расхождение»
  • Много таких закрытий у одного человека за период
  • Короткое время + высокий балл + сложная категория
  • Закрыл сам, не передав целевому
  • Одинаковые формулировки пачкой
MVP баланса
  1. 2–3 явных typical для разбора (расхождение; расследование ЧЗ; при необходимости транзакции).
  2. Список целевых + правило передачи.
  3. В премии полный вес сложных категорий — у целевых / после передачи.
  4. Еженедельный отчёт: кто с конвейера закрыл сложные категории.

Подробно: docs/15-balance-conveyor-vs-investigations.md

Премия, SLA, выезды (кратко)

  • Премия: нормативные баллы в контексте сегмента (конвейер/пул) + качество/аудит позже; не сырая сумма по всей компании
  • Дежурство: ×2 только по графику/табелю, не ручной чекбокс как источник истины
  • Выезды: только 60 при чекбоксе (без суммы с typical); причина выезда для аудита
  • Осложнение: поле-список Okdesk +15 или +30 + текст описания
  • SLA: группа = очередь/владелец смены; человек = после взятия
  • Внеурочка вне смены: пока не выделяем

Как продолжить на другом компьютере

  1. Клонировать https://github.com/averstech2026/okdesk-score-rules.git (если ещё не запушено — закоммитить/запушить этот HTML и docs с текущего ПК).
  2. Открыть этот файл: analysis/typical-catalog.html в браузере.
  3. Читать docs/ по номерам 00→13.
  4. Для новой выгрузки: .env с OKDESK_DOMAIN и OKDESK_API_TOKEN, затем:
    python3 scripts/fetch_issues.py 6
    python3 scripts/analyze_scores.py
  5. В Cursor: открыть папку проекта как workspace (не home), продолжить с плана §1.
