### Вердикт по расходу на премии

- **Базовый прогноз для фонда (замена самооценки на нормы на том же объёме scored):** расход **изменится на +214 350 ₽** за период (~**+30 621 ₽/мес**, +7.3%).
- **MFC техзакрытия без баллов — это норма процесса**, не «дыра». Их **3824**; оплата идёт через дневные компенсаторы («МФС. Заявки, звонки» и аналоги): найдено **118** шт., факт **311 100 ₽**, по норме T5×Nтех/день → **290 550 ₽** (-6.6%).
- Сценарий «нормы на все» после правки MFC ≈ **3 232 125 ₽** (+9.7%) — рост возможен только за счёт **не-MFC** заявок без баллов (400), не за счёт техbulk.

Главные эффекты нового подхода в деньгах:

1. **Пакеты по объектам** (раскатка/обновление/запуск с перечнем касс) считаются как **база × N**, а не срезаются до одной нормы — это не «накрутка», а объём.
2. **Срез** остаётся только у крупных баллов **без** признаков пакета (119 заявок, **322 575 ₽**).
3. **MFC:** техbulk без баллов = 0; компенсатор = **5 × число техзаявок за день**.
4. **Дежурство ×2** в новой модели применяется к *норме*, а не к свободной цифре.
5. Перераспределение между людьми сильнее, чем сдвиг фонда: выигрывают те, у кого много «честных» 5–10 на C15-категориях.

## 1a. Находка по MFC (компенсирующая заявка)

В выгрузке подтверждается правило процесса:

1. Массовые технические заявки MFC часто **без баллов** — это ожидаемо.
2. Почти каждый день есть **одна (или несколько) сводных заявок** вида «МФС. Заявки, звонки» с крупным баллом — компенсация объёма.
3. В модели: 118 компенсаторов, 3824 техbulk обнулены в новом подходе, корреляция числа техзаявок дня с суммой компенсатора на практике умеренная (~0.46), медиана «баллов компенсатора / техзаявка» ≈ 5 → норма T5×N согласована с фактом.

## 1b. Пакетные заявки (N объектов в одной)

Крупный балл (часто ≥100) — как правило **не свободная самооценка**, а одна заявка с перечнем однотипных работ по объектам (обновление ПО, раскатка, запуск, тарифы на кассы…). Оценка должна соответствовать **числу объектов × норма категории**.

В симуляции:

- детект пакета: маркеры в теме (апдейт/раскатка/запуск/…) или ≥3 объекта в описании/таблице, или typical «Доп. настройки»/оборудование при балле ≥100;
- норма = `(база T5/C15/S30 × коэффициент дежурства) × N`;
- N берётся из описания, иначе восстанавливается из факта (`round(факт / единица)`), поэтому пакет почти не «режется».

## 2. Методология

### 2.1. Факт (как в выгрузке)

- Берётся поле `ticket_weight` (баллы, которые поставил инженер).
- Нет баллов → **0 ₽** по этой заявке.
- Множитель дежурства **повторно не начисляется**: в данных медиана на дежурстве уже 10 vs 5 вне дежурства — часть ×2, похоже, уже в выборе балла. Для чувствительности есть сценарий `fact_duty_x2` (если бы ×2 начисляли сверху на уже проставленный балл).
- Премия ≈ баллы × 15 ₽.

### 2.2. Новый подход (симуляция)

Формула из принципов:

```
если выезд: итого = 60 × (2 если дежурство else 1)
иначе:     итого = (база × N + осложнение) × (2 если дежурство else 1)
```

| Элемент | Как смоделировано |
|---------|-------------------|
| База | `typical` → T5=5 / C15=15 / S30=30 (таблица ниже) |
| Пустой `typical` | T5=5 (консервативно; **не** для MFC-техbulk) |
| «Другое» | C15=15 (пока нет «Нестандарт» + дыр каталога); **не** свободные 500–1000 |
| **Пакет по объектам** (раскатка/апдейт/запуск/…) | `база × N` объектов; N из описания или из факта |
| **MFC техbulk** (без баллов, не компенсатор) | **0** — оплата через дневной пакет |
| **MFC компенсатор** (заголовок «…Заявки…», клиент MFC) | `5` × число техзаявок MFC за тот же день (`MFC_DAY`) |
| Выезд (`departure`) | **только 60** (typical / N / осложнение не суммируются) |
| Осложнение | уровни **+15** / **+30** + текст (в базовой симуляции: X_LONG≈+15 при ≥2 ч) |
| Дежурство (`watch`) | ×2 к сумме нормы |
| Осложнение `X_LONG` (сценарий) | +15, если `spent_time_total` ≥ 2 ч |
| Гейт расследований (сценарий) | `Расхождение…` = S30 только целевым; иначе T5 (черновой список целевых) |
| Сегменты премии (конвейер/пул) | **не меняют рубли в этой таблице**, но меняют интерпретацию «кто молодец» — см. § 7 |

## 7. Что это значит для расхода на премии

### 7.1. Три развилки решения

| Решение | Влияние на ФОТ | Комментарий |
|---------|----------------|-------------|
| Нормы + правило MFC (техbulk=0, компенсатор=T5×N) | ФОТ → ~3 232 125 ₽ (+9.7%) | **Базовая модель этого отчёта** — отражает текущий процесс MFC |
| Полностью вынести MFC из балльной премии | ориентир ~2 411 775 ₽ на не-MFC | Только если компенсатор/пакет MFC оплачивается иначе |
| Сначала заменить только самооценку там, где баллы уже ставят | ~3 160 275 ₽ (+7.3%) | Близко к new_all после учёта MFC; удобно для теневого периода |

### 7.2. Ожидания как руководителю

- На **сопоставимом объёме** (scored) модель даёт **рост** фонда примерно на **214 350 ₽** (+7.3%) за период.
- После учёта компенсатора MFC сценарий «новый все» ≈ **3 232 125 ₽** (+9.7%) — техbulk больше не раздувает фонд.
- **Болезненность точечная:** сильнее всего у тех, кто закрывал массовые раскатки/запуски крупными цифрами при коротком времени; у «честного» конвейера 5–10 изменение мягкое или в плюс (C15 на консультации/доп.настройки). Дневной пакет MFC при норме T5×N близок к текущей медиане.
- **Политика сегментов** (конвейер vs пул vs расследования) почти не меняет сумму в рублях в этой симуляции, но меняет справедливость внутри команды — без неё формально «правильные» баллы всё равно будут толкать людей на удобный поток.
- Дежурства: при переходе на ×2 от **нормы** по табелю (не от самооценки) ожидайте пересчёт дежурного контура 1,5 недели отдельно от месячного.

### 7.3. Рекомендуемый порядок внедрения (чтобы не ударить ФОТ и людей)

1. Утвердить маппинг typical→база и **явный список конвейерных клиентов** (IFCM, MFC, …).
2. MFC: зафиксировать в регламенте «техbulk без баллов + дневной компенсатор»; в нормах — `MFC_DAY = T5 × N` (или иная ставка), не начислять баллы на каждую техзаявку.
3. Включить нормы на нативном контуре; параллельно 1–2 месяца **теневой расчёт** (как этот отчёт) без смены выплат.
4. Срезать свободный select; крупные работы (раскатка/запуск) — отдельные категории с нормой + аудит, не 500–1000 «от себя».
5. Потом — трек расследований и гейт целевых (docs/15).

## 8. Ограничения модели

- Каталог T5/C15/S30 — **черновик**; смена консультации с факт-медианы 5 на C15=15 заметно двигает фонд.
- Выезд: только +60 при `departure` (без суммы с typical).
- Осложнения почти не восстановить из API → `X_LONG` грубый.
- Список целевых для расследований — **заглушка** для чувствительности, не HR-решение.
- Не моделировались качество, SLA-после-взятия, срезание по аудиту.
- Часть заявок без `completed_at` могла уйти в `updated_at` (см. скрипт).
